1 ภาพรวม & เป้าหมายธุรกิจ

แพลตฟอร์มสะสมแต้มหน้าร้านแบบหลายแบรนด์ (SaaS) — พนักงานทำงานผ่าน Staff Terminal ลูกค้าระบุตัวด้วยเบอร์

แพลตฟอร์มสะสมแต้มแบบ multi-tenant (SaaS) ที่ให้หลายแบรนด์/ร้านเปิดใช้บนระบบเดียว แต่ละแบรนด์ตั้งกฎสะสมแต้ม รางวัล ระดับสมาชิก และแคมเปญของตัวเองได้ การสะสม/แลกแต้มทั้งหมดทำที่หน้าร้านผ่าน Staff Terminal (เว็บ responsive) — พนักงานค้นหา/สมัครสมาชิกด้วยเบอร์โทร บันทึกยอดซื้อเพื่อคำนวณแต้ม และแลกรางวัลให้ลูกค้า ลูกค้าปลายทางไม่มีหน้าใช้เอง รองรับ Tier, แคมเปญแต้มคูณ และแนะนำเพื่อน · เริ่มแบบ standalone ยังไม่ต่อ POS/payment เดิม

  • ปัญหาที่แก้: ให้หลายแบรนด์ทำโปรแกรมสะสมแต้ม/ลอยัลตี้หน้าร้านได้เองบนระบบเดียว โดยไม่ต้องลงทุนระบบสมาชิกแยกของใครของมัน พนักงานใช้งานง่ายจบในจอเดียว ลูกค้ากลับมาซื้อซ้ำเพราะมีแต้ม/รางวัล/ระดับสมาชิก และแบรนด์มีข้อมูลพฤติกรรมไว้ทำการตลาด
  • ตัววัดความสำเร็จ: จำนวนสมาชิก active, อัตราการกลับมาซื้อซ้ำ (repeat rate), แต้มที่ออก/ถูกแลก (redemption rate), จำนวนสมาชิกที่เลื่อน tier, จำนวนสมาชิกที่มาจาก referral, จำนวนแบรนด์ที่ใช้งานจริง

2 ขอบเขตงาน (Scope)

แยก in/out ให้ชัด — กันงานบานปลายและความเข้าใจผิดเรื่องราคา

✅ อยู่ในขอบเขต

  • ระบบ multi-tenant: สมัคร/เปิด-ปิดแบรนด์, จัดการสาขา, แยกข้อมูลแต่ละแบรนด์ (tenant isolation)
  • Auth & สิทธิ์ฝั่งแบรนด์ (เจ้าของ / พนักงาน) + Platform Admin
  • ทะเบียนสมาชิก: สมัคร/ค้นหาด้วยเบอร์โทร, โปรไฟล์, ประวัติแต้ม/การแลก (กันสมาชิกซ้ำ)
  • เครื่องมือสะสมแต้ม (Points Engine): กฎ earn ตามยอดซื้อ, บันทึกยอดซื้อ (คีย์มือ), points ledger, ปรับแต้มมือ, หมดอายุแต้ม
  • แลกรางวัล + คลังรางวัล/สต็อก (ตัดแต้ม-ตัดสต็อก, ประวัติการแลก)
  • Staff Terminal หน้าร้าน (เว็บ responsive): ค้น/สมัครสมาชิก, สะสม, แลก จบในจอเดียว
  • ระดับสมาชิก (Tier): เกณฑ์เลื่อน/ตกชั้น, ตัวคูณแต้ม, สิทธิ์ต่อ tier
  • โปรโมชัน/แคมเปญ: แต้มคูณ x2, แต้มโบนัสตามช่วงเวลา/สินค้า/วันเกิด, เงื่อนไขแคมเปญ
  • แนะนำเพื่อน (Referral): รหัส/เบอร์ผู้แนะนำ, ให้แต้มทั้งสองฝ่าย, กันโกง
  • Dashboard & รายงานฝั่งแบรนด์ + Dashboard ฝั่ง Platform Admin

⛔ อยู่นอกขอบเขต

  • หน้า/แอป self-service ฝั่งลูกค้า (ดูแต้มเอง, แลกเอง ผ่าน LINE/เว็บ/มือถือ) — ทำผ่านพนักงานเท่านั้น
  • เชื่อม POS / payment / ERP / CRM เดิมของร้าน (คีย์ยอดซื้อมือในเฟสนี้)
  • ระบบเก็บค่าบริการแบรนด์ (billing/subscription) & ชำระเงิน — เสนอแยกเมื่อยืนยันโมเดลรายได้
  • แจ้งเตือนภายนอก (SMS/LINE/Email) — เสนอแยกเมื่อต้องการ
  • การออกบัตรสมาชิกจริง/บัตรพลาสติก, การพิมพ์การ์ด
  • เนื้อหา/รูปรางวัล & สื่อการตลาด (แบรนด์จัดเตรียมเอง)
  • ค่า cloud/hosting/โดเมน และงานดูแลหลังส่งมอบ (เสนอแยก)

3 ผู้ใช้ & บทบาท (Actors)

บทบาททำอะไรได้
ลูกค้า / สมาชิก (หน้าร้าน)ระบุตัวด้วยเบอร์โทรที่หน้าร้าน, สะสมแต้มจากยอดซื้อ, แลกรางวัล, เลื่อนระดับสมาชิก, รับแต้มจากแคมเปญ/การถูกแนะนำ — ทั้งหมดทำผ่านพนักงาน (ไม่มี self-service)
พนักงานหน้าร้าน (Staff)ค้นหา/สมัครสมาชิกด้วยเบอร์, บันทึกยอดซื้อเพื่อสะสมแต้ม, แลกรางวัล, กรอกรหัส/เบอร์ผู้แนะนำตอนสมัคร, ดูแต้ม/tier/ประวัติของสมาชิก — ผ่าน Staff Terminal
เจ้าของแบรนด์ / ร้าน (Merchant)ตั้งกฎสะสมแต้ม & เงื่อนไขหมดอายุ, จัดการคลังรางวัล, ตั้ง Tier & สิทธิ์, สร้างแคมเปญ/โปรโมชัน, ตั้งกติกา referral, จัดการสาขา/พนักงาน, ดูรายงานแบรนด์
ผู้ดูแลแพลตฟอร์ม (Platform Admin)อนุมัติ/เปิด-ปิดแบรนด์ (tenant), ดูภาพรวมการใช้งานทุกแบรนด์, จัดการสิทธิ์ระดับแพลตฟอร์ม

7 Timeline & เปรียบเทียบ Stack

ทีม freelance 2–3 คน (full-stack 2 + PM/QA แชร์ · ~2 effective heads)

Effort รวม ~138–232 man-day → ระยะเวลาจริงราว 4–7 เดือน

▲ Laravel (PHP) + Filament

⏱ ~4–6 เดือน

  • งานนี้เป็น admin/CRUD + รายงานเป็นหลัก (ไม่มี self-service ลูกค้า) — Filament scaffold กลุ่ม A/C/D ได้เปรียบมาก
  • เหมาะ multi-tenant + report + points ledger
  • หา dev ไทยง่าย
  • Staff Terminal ต้องทำ frontend ให้ลื่นเองบ้าง

▲ Next.js (React/TS)

⏱ ~4.5–7 เดือน

  • Staff Terminal เป็น PWA ลื่น ใช้หน้าร้านดี
  • UX มือถือ/แท็บเล็ตทันสมัย
  • ecosystem JS เดียวจบ
  • งาน admin/รายงาน/แคมเปญ CRUD ต้องประกอบเองมากกว่า — เสียเวลากลุ่ม A/C/D
สรุป: งานนี้น้ำหนักไปทางจัดการ/รายงาน (A/C/D) เพราะไม่มีหน้า self-service ลูกค้า — Laravel + Filament ได้เปรียบด้าน time-to-market ชัดกว่า coupon ที่มี PWA ลูกค้า · ถ้าทีมถนัด JS ใช้ Next.js ก็ได้ แต่ต้องเผื่อเวลากลุ่ม admin มากขึ้น

  • งาน integration/edge case/R&D ช้ากว่าคาด — ตั้งตัวเร่งอนุรักษ์นิยม (2–3x) + contingency 25%
  • คุณภาพขึ้นกับ skill ผู้ควบคุม AI — ใช้ dev senior ขับ + กำหนดมาตรฐานรีวิว/เทสต์

9 สมมติฐาน · ความเสี่ยง · คำถามที่ต้องถามเพิ่ม

สมมติฐาน (Assumptions)

  • ลูกค้าปลายทางไม่มี self-service — ทุกการสะสม/แลกทำผ่านพนักงานที่ Staff Terminal (เว็บ responsive); ระบุตัวสมาชิกด้วยเบอร์โทร
  • Referral ในบริบท POS-only = พนักงานกรอกรหัส/เบอร์ผู้แนะนำตอนสมัครสมาชิกใหม่ (ไม่มี channel ให้ลูกค้าแชร์เอง)
  • บันทึกยอดซื้อด้วยการคีย์มือที่ terminal เพื่อคำนวณแต้ม — ไม่ต่อ POS/payment เดิมในเฟสนี้
  • เริ่มแบบ standalone ไม่มี integration ภายนอก (SMS/LINE/Email/POS/bank) ในราคานี้
  • เก็บข้อมูลสมาชิกขั้นต่ำ (ชื่อ/เบอร์) ตอนสมัคร — ต้องมี consent ตาม PDPA ที่หน้าร้าน
  • scale กลาง: single DB แบบ row-level multi-tenant (tenant_id) บน cloud มาตรฐานเพียงพอ
  • โมเดลเก็บค่าบริการแบรนด์ (billing) ยังไม่อยู่ใน scope รอบนี้ — ถ้าทำในอนาคตจะใช้ค่าบริการ SaaS ปกติ ไม่มีดอกเบี้ย (halal)
  • ใช้ design system / component library สำเร็จรูป ไม่ออกแบบ UI จากศูนย์
  • พนักงานใช้เว็บ responsive บนมือถือ/แท็บเล็ต ไม่ต้องมี native app แยก

ความเสี่ยง (Risks)

ความเสี่ยงผลกระทบโอกาสแนวทางรับมือ
พนักงานคีย์ยอดซื้อ/เลือกสมาชิกผิด → แต้มเข้าผิดคนหรือผิดจำนวนกลางกลางยืนยันก่อนบันทึก, แสดงชื่อ/เบอร์ชัด, ปรับแต้มย้อนได้ตามสิทธิ์ + เก็บ log ทุกครั้ง
โกงแต้ม/referral (สมัครซ้ำเอาแต้มแนะนำ, ปั๊มยอด)กลางกลางdedupe เบอร์, จำกัดจำนวน referral, เงื่อนไขปลดแต้ม, log + รายงานผิดปกติ
กฎแต้ม/tier/แคมเปญซับซ้อนและเปลี่ยนบ่อยกลางสูงออกแบบ engine แบบ config-driven, นิยามลำดับการคิดแต้มเมื่อหลายแคมเปญทับซ้อนให้ชัดแต่ต้น
ข้อมูลข้ามแบรนด์รั่ว (tenant isolation)สูงต่ำบังคับ tenant scoping ทุก query + เทสต์เฉพาะ + code review
สมาชิกซ้ำ/หาไม่เจอ (เบอร์เปลี่ยน, สะกดผิด)กลางกลางค้นหลาย field, ฟังก์ชัน merge สมาชิก, ตรวจซ้ำตอนสมัคร

คำถามที่ต้องถามลูกค้าเพิ่ม

  • กฎสะสมแต้ม: ซื้อครบเท่าไรได้กี่แต้ม? คิดตามยอดซื้อรวมหรือต่อบิล? มีปัดเศษอย่างไร?
  • แต้มหมดอายุไหม และนานเท่าไร (เช่น 1 ปีนับจากได้รับ หรือสิ้นปี)?
  • Tier มีกี่ระดับ เกณฑ์เลื่อน/ตกชั้นอย่างไร และสิทธิ์/ตัวคูณแต่ละระดับเป็นอย่างไร?
  • Referral ให้แต้มผู้แนะนำ/ผู้ถูกแนะนำฝ่ายละเท่าไร และมีเงื่อนไขปลดแต้ม/กันโกงแบบไหน?
  • บันทึกยอดซื้อยังไงในทางปฏิบัติ (คีย์มือ vs สแกนใบเสร็จ vs อนาคตต่อ POS)?
  • คาดว่ามีกี่แบรนด์/สาขา/สมาชิก และทรานแซกชันสะสม-แลกต่อวันประมาณเท่าไร (นิยาม 'กลาง')?
  • โมเดลรายได้ของแพลตฟอร์มเป็นอย่างไร (เก็บค่าบริการแบรนด์ไหม) — มีผลต่อเฟสถัดไป?
  • อนาคตต้องการให้ลูกค้ามีหน้าดูแต้ม/แลกเอง (LINE/เว็บ) หรือแจ้งเตือนแต้มไหม?

สิ่งที่ไม่รวมในราคานี้

  • ค่า cloud / hosting / โดเมน
  • ค่า SMS / LINE / Email notify (หากเพิ่มภายหลัง)
  • งานเชื่อม POS / payment / bank (เสนอแยกเมื่อต้องการ)
  • ระบบเก็บค่าบริการแบรนด์ (billing/subscription) — เสนอแยก
  • เนื้อหา / รูปรางวัล & สื่อการตลาด (แบรนด์จัดเตรียม)
  • บัตรสมาชิกพลาสติก / การพิมพ์การ์ด
  • งานดูแลหลังส่งมอบ (maintenance) — เสนอแยกเป็นสัญญารายปี
