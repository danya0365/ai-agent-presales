1 ภาพรวม & เป้าหมายธุรกิจ

แพลตฟอร์มคูปองหน้าร้านแบบหลายร้าน (SaaS) — ลูกค้าสแกนแล้วใช้ ร้านจ่ายค่าบริการรายวัน

แพลตฟอร์มคูปองแบบ multi-tenant (SaaS) ที่ให้ร้านค้าหลายร้านเปิดใช้บนระบบเดียว แต่ละร้านสร้างคูปองส่วนลด/โปรโมชัน และคูปองแลกของรางวัลของตัวเองได้ ลูกค้าหน้าร้านสแกน QR เพื่อรับ/ดูคูปองของตัวเอง (ผูกกับเครื่อง ไม่ต้อง login) แล้ว redeem ที่หน้าร้าน เจ้าของแพลตฟอร์มคิดค่าบริการรายวันต่อร้าน ชำระผ่าน PromptPay

  • ปัญหาที่แก้: ให้ร้านค้าทำแคมเปญคูปอง/ของรางวัลหน้าร้านได้เองโดยไม่ต้องมีระบบสมาชิกซับซ้อน ลูกค้าเข้าถึงคูปองง่าย (สแกนแล้วใช้) และเจ้าของแพลตฟอร์มมีรายได้แบบ subscription รายวันจากหลายร้าน
  • ตัววัดความสำเร็จ: จำนวนร้านที่ active, จำนวนคูปองที่ออก/ถูกใช้ (redemption rate), ยอดของรางวัลที่แลก, รายได้ค่าบริการรายวันรวม, อัตราร้านที่ต่ออายุ

2 ขอบเขตงาน (Scope)

แยก in/out ให้ชัด — กันงานบานปลายและความเข้าใจผิดเรื่องราคา

✅ อยู่ในขอบเขต

  • ระบบ multi-tenant: สมัคร/เปิด-ปิดร้าน, แยกข้อมูลแต่ละร้าน (tenant isolation)
  • Auth & สิทธิ์ฝั่งร้าน (เจ้าของร้าน / พนักงาน)
  • จัดการคูปองส่วนลด/โปรโมชัน (%, จำนวนเงิน, ซื้อ 1 แถม 1, เงื่อนไข, วันหมดอายุ, จำกัดจำนวน)
  • คูปองแลกของรางวัล + คลังของรางวัล/สต็อก
  • QR generation + device-bound session ฝั่งลูกค้า (ไม่ต้อง login, ผูกร้าน)
  • Redemption หน้าร้าน (สแกน/ยืนยัน, กันใช้ซ้ำ, log)
  • ระบบคิดค่าบริการรายวันต่อร้าน + รอบบิล + auto-suspend เมื่อค้างชำระ
  • ชำระเงิน PromptPay (สร้าง QR EMVCo + อัปสลิป + แอดมินยืนยัน manual)
  • Dashboard & รายงานฝั่งร้าน + Dashboard ฝั่ง Platform Admin

⛔ อยู่นอกขอบเขต

  • การ reconcile PromptPay อัตโนมัติ (ต่อ bank API / payment gateway)
  • ระบบสมาชิก/บัญชีลูกค้าแบบเต็ม (login, ผูกเบอร์/อีเมล) — ใช้ device session แทน
  • การใช้/แลกคูปองออนไลน์นอกหน้าร้าน, e-commerce, ตะกร้าสินค้า
  • เชื่อม POS / ERP / CRM / ระบบบัญชีเดิมของร้าน
  • เนื้อหา/รูปคูปองและของรางวัล (ร้านจัดเตรียมเอง)
  • ค่า cloud/hosting/โดเมน, ค่าธรรมเนียมธนาคาร, งานดูแลหลังส่งมอบ (เสนอแยก)

3 ผู้ใช้ & บทบาท (Actors)

บทบาททำอะไรได้
ลูกค้า / ผู้ถือคูปอง (หน้าร้าน)สแกน QR ที่ร้านเพื่อรับ/ดูคูปองของตัวเอง (เก็บ session ในเครื่อง), ใช้คูปองส่วนลด, แลกของรางวัล — เฉพาะร้านนั้น
พนักงานร้าน (Staff)สแกน/ยืนยันการใช้คูปองหน้าร้าน, ออก QR ให้ลูกค้า, ค้นหาลูกค้าเดิมแล้วผูกกับเครื่องใหม่, ตรวจของรางวัลคงเหลือ
เจ้าของร้าน (Merchant)สร้าง/จัดการคูปอง & ของรางวัล, ดูรายงานร้าน, ชำระค่าบริการรายวัน, จัดการพนักงาน
ผู้ดูแลแพลตฟอร์ม (Platform Admin)อนุมัติ/เปิด-ปิดร้าน, ดูบิล & รายได้รวม, ยืนยันการชำระ PromptPay, ดูภาพรวมทุกร้าน

▲ Laravel (PHP) + Filament

⏱ ~3.5–5.5 เดือน

  • Admin/CRUD ร้าน + billing + dashboard scaffolding เร็ว (กลุ่ม A/C/D ได้เปรียบ)
  • โตเต็มที่ เหมาะ multi-tenant + รายงาน
  • หา dev ไทยง่าย
  • ส่วน QR/PWA device session ฝั่งลูกค้าต้องทำ frontend เพิ่มเอง

▲ Next.js (React/TS)

⏱ ~4–6 เดือน

  • PWA/device session ฝั่งลูกค้า + สแกน QR ลื่น (กลุ่ม B ได้เปรียบ)
  • UX มือถือทันสมัย เหมาะหน้าร้าน
  • ecosystem JS เดียวจบ
  • งาน admin/billing CRUD ต้องประกอบเองมากกว่า
สรุป: งานนี้แบ่งเป็นสองฝั่งชัด — ฝั่งจัดการ/บิล (A/C/D) Laravel เร็วกว่า ส่วนฝั่งลูกค้า/QR (B) Next.js เร็วกว่า ผลต่างรวมเล็ก (~10%) เลือกตามทีมที่หาได้จริง หรือทำ hybrid (Laravel API + PWA frontend) ก็ได้

สมมติฐานเฉพาะโหมด AI

  • ขับด้วย dev ที่ skill สูง 1 คนต่อ 1 สายงาน + AI coding agent (เช่น Claude Code)
  • โค้ดที่ AI สร้างผ่านรีวิว + เทสต์โดยคนก่อนส่งมอบ (วันรีวิว/เทสต์รวมในตัวเลขแล้ว)
  • งานออกแบบ/ตัดสินใจ/ความปลอดภัย/integration/R&D AI เร่งได้น้อยกว่า — สะท้อนในตัวเร่งแล้ว
  • ขอบเขต/โมดูลเท่าฐานทีมคนเป๊ะ (เทียบ apples-to-apples) เรตเท่าเดิม

ความเสี่ยงเฉพาะโหมด AI

  • AI สร้างโค้ดผิด/หลอน (hallucination) — รีวิว+เทสต์โดยคนทุกงาน วันรีวิวรวมในตัวเลขแล้ว
  • งาน integration/edge case/R&D ช้ากว่าคาด — ตั้งตัวเร่งอนุรักษ์นิยม (2–3x) + contingency 25%
  • คุณภาพขึ้นกับ skill ผู้ควบคุม AI — ใช้ dev senior ขับ + กำหนดมาตรฐานรีวิว/เทสต์

9 สมมติฐาน · ความเสี่ยง · คำถามที่ต้องถามเพิ่ม

สมมติฐาน (Assumptions)

  • ลูกค้าปลายทางไม่มีระบบ login เต็ม — คูปองเก็บฝั่ง server ผูกกับทะเบียนลูกค้าเบื้องต้น (ชื่อ/เบอร์) ของแต่ละร้าน; session เป็น device-bound token (localStorage/PWA); เปลี่ยนเครื่อง = พนักงานค้นหาลูกค้าแล้วผูกเครื่องใหม่ คูปองไม่หาย
  • เก็บข้อมูลลูกค้าขั้นต่ำ (ชื่อ/เบอร์) ตอนสแกนครั้งแรกเพื่อให้พนักงานค้นหา/ผูกเครื่องใหม่ได้ — ต้องมี consent ตาม PDPA ที่หน้าร้าน
  • PromptPay สร้าง QR เอง (EMVCo) ไม่ต่อ bank API → ยืนยันยอดชำระแบบ manual (อัปสลิป + แอดมินยืนยัน)
  • ค่าบริการรายวันเป็นค่า subscription SaaS ปกติ ไม่มีดอกเบี้ย (halal)
  • scale กลาง: ใช้ single DB แบบ row-level multi-tenant (tenant_id) บน cloud มาตรฐานเพียงพอ
  • redeem ที่หน้าร้านเท่านั้น ไม่มีการใช้คูปองออนไลน์
  • ใช้ design system / component library สำเร็จรูป ไม่ออกแบบ UI จากศูนย์
  • พนักงานร้านสแกนผ่านเว็บ responsive บนมือถือ ไม่ต้องมี native app แยก

ความเสี่ยง (Risks)

ความเสี่ยงผลกระทบโอกาสแนวทางรับมือ
พนักงานค้นหา/ผูกลูกค้าผิดคนตอนเปลี่ยนเครื่อง → คูปองไปเข้าผิดคนกลางกลางค้นด้วยเบอร์+ชื่อ, ยืนยันตัวตนก่อนผูก, เก็บ log ทุกการผูกเครื่อง
ยืนยัน PromptPay แบบ manual → งานแอดมินเยอะ/ผิดพลาดกลางกลางออกแบบ flow อัปสลิป + กระทบยอดง่าย, เผื่อต่อ bank API ภายหลัง
ข้อมูลข้ามร้านรั่ว (tenant isolation)สูงต่ำบังคับ tenant scoping ทุก query + เทสต์เฉพาะ, code review
ปลอม QR / ใช้คูปองซ้ำสูงกลางsigned token + ตรวจสิทธิ์ server-side + redemption idempotent
ร้านค้างชำระค่าบริการรายวันกลางกลางauto-suspend + แจ้งเตือนล่วงหน้า + ปลดล็อกเมื่อชำระ

คำถามที่ต้องถามลูกค้าเพิ่ม

  • ค่าบริการรายวันต่อร้านคิดเท่าไร และมีแพ็กเกจ/tier หลายระดับไหม?
  • เก็บข้อมูลลูกค้าแค่ไหนตอนสแกนครั้งแรก (ชื่อ/เบอร์อย่างเดียว?) และวาง consent PDPA ที่หน้าร้านอย่างไร?
  • คาดว่ามีร้านกี่ร้าน และทรานแซกชัน redeem ต่อวันประมาณเท่าไร (นิยาม 'กลาง')?
  • ต้องมีแอป staff แยกไหม หรือเว็บ responsive สแกนผ่านมือถือพอ?
  • อนาคตต้องการ reconcile PromptPay อัตโนมัติ (ต่อ bank/gateway) ไหม?
  • ต้องออกใบเสร็จ/ใบกำกับภาษีค่าบริการให้ร้านไหม?
  • คูปองผูกร้านเดียวเท่านั้น หรือมีแคมเปญข้ามร้าน?

สิ่งที่ไม่รวมในราคานี้

  • ค่า cloud / hosting / โดเมน
  • ค่าธรรมเนียมธนาคาร / PromptPay (ถ้ามี)
  • ค่า SMS / LINE notify (หากเพิ่มภายหลัง)
  • เนื้อหา / รูปคูปอง & ของรางวัล (ร้านจัดเตรียม)
  • งานดูแลหลังส่งมอบ (maintenance) — เสนอแยกเป็นสัญญารายปี
