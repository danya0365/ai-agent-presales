1 ภาพรวม & เป้าหมายธุรกิจ

แพลตฟอร์มกลางของกรม — มองตั้งแต่คลังสื่อจนถึง AI แนะนำการเรียนรู้

แพลตฟอร์มกลางของกรมส่งเสริมการเรียนรู้ ทำหน้าที่เป็นคลังหนังสืออิเล็กทรอนิกส์ (e-book PDF) + ระบบสั่งซื้อ/เบิกสื่อสำหรับโรงเรียนและผู้เรียนในสังกัด แล้วต่อยอดเชื่อมเข้าระบบกลาง (SSO, Big Data) จนถึง AI แนะนำการเรียนรู้รายบุคคล

  • ปัญหาที่แก้: รวมศูนย์สื่อการเรียนรู้ดิจิทัลของกรมไว้ที่เดียว ให้โรงเรียนเข้าถึง/เบิก/สั่งซื้อสะดวก ลดงานเอกสาร และมีข้อมูลการใช้งานเพื่อวางนโยบาย
  • ตัววัดความสำเร็จ: จำนวนสื่อในคลัง, ยอดดาวน์โหลด/สั่งซื้อ, จำนวนโรงเรียน/ผู้ใช้ active, ความครอบคลุมรายจังหวัด

2 ขอบเขตงาน (Scope)

แยก in/out ให้ชัด — กันงานบานปลายและความเข้าใจผิดเรื่องราคา

✅ อยู่ในขอบเขต

  • ระบบจัดการสิทธิ์ผู้ใช้ & บทบาท (RBAC)
  • คลังหนังสืออิเล็กทรอนิกส์ + อัปโหลด/จัดหมวด PDF + PDF viewer
  • API เชื่อมข้อมูลหนังสือจากโรงพิมพ์
  • ระบบสั่งซื้อ/เบิกสื่อ + ประวัติการสั่งซื้อ
  • รายงาน & สถิติ (ยอดดาวน์โหลด, คำสั่งซื้อรายจังหวัด)
  • เชื่อม Single Sign-On (SSO) กับ identity กลางกรม
  • Data pipeline เข้าสู่ Big Data + dashboard วิเคราะห์
  • AI Adaptive Learning (เชื่อม Claude/Gemini/OpenAI + แนะนำสื่อรายบุคคล)

⛔ อยู่นอกขอบเขต

  • เนื้อหา/ลิขสิทธิ์หนังสือ (ลูกค้า/โรงพิมพ์จัดเตรียม)
  • ค่า license/โทเคน AI รายเดือน (Claude/Gemini/OpenAI)
  • ระบบบัญชี/การเงินภายในกรม
  • การพัฒนาระบบปลายทางที่เชื่อม (HRMS, E-Saraban, EKAS ฯลฯ) — ทำเฉพาะส่วนเชื่อมต่อฝั่งเรา
  • Migrate ข้อมูลเก่า หากยังไม่ระบุปริมาณ/รูปแบบ
  • ค่า cloud/hosting/โดเมน & งานดูแลหลังส่งมอบ (เสนอแยก)

7 Timeline & เปรียบเทียบ Stack

ทีม freelance 3 คน (full-stack 2 + AI/Data 1, PM/QA แชร์ · ~2.5 effective heads)

Effort รวม ~220–380 man-day → ระยะเวลาจริงราว 5–9 เดือน

▲ Next.js (React/TS)

⏱ ~5.5–9 เดือน

  • เชื่อม AI/real-time ลื่น (อยู่ใน ecosystem JS เดียวกัน)
  • UX/SPA ทันสมัย เหมาะ dashboard + viewer
  • หา dev รุ่นใหม่ง่าย
  • งาน CRUD/admin ต้องประกอบเองมากกว่า

▲ Laravel (PHP)

⏱ ~5–8.5 เดือน

  • CRUD/admin scaffolding เร็ว (Filament) → กลุ่ม A เร็วกว่า ~10%
  • โตเต็มที่ เหมาะ order/report/admin-heavy
  • ทีมภาครัฐไทยคุ้น PHP
  • AI/Big Data ยังต้องพึ่ง service Python แยก
สรุป: ผลต่างเวลา/ราคาระหว่างสอง stack เล็กมาก (~5–8%) เพราะกลุ่ม A ที่ Laravel ได้เปรียบเป็นแค่ ~1/3 ของงาน ส่วน AI/Big Data ต้องใช้ Python เหมือนกันทั้งคู่ → เลือกตามทีมที่หาได้จริงดีกว่าเลือกตามตัวเลข

9 สมมติฐาน · ความเสี่ยง · คำถามที่ต้องถามเพิ่ม

สมมติฐาน (Assumptions)

  • กรมมีระบบ identity กลางสำหรับ SSO พร้อมเอกสาร API ให้เชื่อม
  • โรงพิมพ์/หน่วยงานภายนอกมี API หรือไฟล์รูปแบบมาตรฐานให้ดึงข้อมูลหนังสือ
  • ปริมาณผู้ใช้/ไฟล์ PDF อยู่ในระดับที่ใช้สถาปัตยกรรม cloud มาตรฐานรองรับได้
  • ใช้ design system / component library สำเร็จรูป ไม่ออกแบบ UI จากศูนย์
  • ขอบเขต AI = แนะนำสื่อ/เส้นทางเรียนรู้ ไม่ใช่ train โมเดลเอง (เรียกใช้ผ่าน API)

ความเสี่ยง (Risks)

ความเสี่ยงผลกระทบโอกาสแนวทางรับมือ
AI/Big Data requirement ยังไม่นิ่ง (R&D)สูงสูงแยกเป็นเฟส 2, ทำ POC ก่อนล็อกราคา
จำนวน/ความพร้อมของ API ภายนอก (M-07) ไม่ชัดสูงกลางตีราคาเป็นช่วงกว้าง, คิดเป็นรายจุดเชื่อม
การจัดซื้อภาครัฐ + เอกสาร/ขั้นตอนเยอะกลางสูงเผื่อ buffer PM, วาง milestone ชัด
ปริมาณ PDF/ทราฟฟิกโตเกินคาดกลางกลางใช้ object storage + CDN, ออกแบบเผื่อ scale

คำถามที่ต้องถามลูกค้าเพิ่ม

  • จำนวนผู้ใช้/โรงเรียน, ปริมาณ PDF และทราฟฟิกที่คาดไว้?
  • SSO ใช้มาตรฐานอะไร (OIDC/SAML/อื่น ๆ) และมีเอกสารพร้อมหรือยัง?
  • ต้องเชื่อม API ภายนอกกี่ระบบ และระบบไหนพร้อมในเฟสนี้บ้าง?
  • มีข้อมูลเดิมต้อง migrate ไหม ปริมาณเท่าไร รูปแบบใด?
  • ขอบเขต AI Adaptive Learning ที่ต้องการในเฟสแรกแค่ไหน (แนะนำสื่อ / ทำนายผล / chatbot)?
  • งบประมาณกรอบและเดดไลน์ของโครงการ?

สิ่งที่ไม่รวมในราคานี้

  • ค่า cloud / hosting / โดเมน
  • ค่า license & โทเคน AI รายเดือน (Claude / Gemini / OpenAI)
  • ค่า payment gateway / Digital Wallet fee
  • เนื้อหา / ลิขสิทธิ์หนังสือ
  • งานดูแลหลังส่งมอบ (maintenance) — เสนอแยกเป็นสัญญารายปี
