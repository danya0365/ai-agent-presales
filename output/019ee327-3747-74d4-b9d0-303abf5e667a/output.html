1 ภาพรวม & วิเคราะห์ระบบ

รูปทั้ง 3 หน้าที่พี่ส่งมาคือ "ระบบเดียวกัน" — มองจากคนละมุม

มันคือระบบอะไร? เป็นแพลตฟอร์มกลางของกรมส่งเสริมการเรียนรู้ ชื่อ "DOLE Book Store / ระบบจัดการสื่อการเรียนรู้" ทำหน้าที่เป็น คลังหนังสืออิเล็กทรอนิกส์ (e-book PDF) + ระบบสั่งซื้อ/เบิกสื่อ สำหรับโรงเรียนและผู้เรียนในสังกัด แล้วต่อยอดเชื่อมเข้าสู่ระบบกลางของกรม (SSO, Big Data) จนถึง AI แนะนำการเรียนรู้รายบุคคล

รูปหัวข้อในเอกสารบอกอะไรกับเรา
รูปที่ 17.1.3 ค่าใช้จ่ายรายฟังก์ชันระบุ "หัวใจ" ที่ต้องคิดเงิน 5 โมดูล: จัดการสิทธิ์ผู้ใช้, จัดการ e-book, สั่งซื้อ, ประวัติสั่งซื้อ, รายงาน/สถิติ
รูปที่ 27.1.1 Diagram สถาปัตยกรรมภาพใหญ่ทั้ง ecosystem: SSO, DOLE Learner Data, e-Market, Digital Wallet/EKAS/Credit Bank/e-Portfolio, Big Data, AI Adaptive Learning (Claude/Gemini/OpenAI), เชื่อม HRMS/E-Saraban แบ่งเฟส 1–2
รูปที่ 37.1.2 แนวคิดการออกแบบยืนยันชื่อระบบหลัก = DOLE Book Store + รายการระบบงานย่อยและ Single Sign-On

เป้าหมายธุรกิจ (Business Goal)

  • ปัญหาที่แก้: รวมศูนย์สื่อการเรียนรู้ดิจิทัลของกรมไว้ที่เดียว ให้โรงเรียนเข้าถึง/เบิก/สั่งซื้อได้สะดวก ลดงานเอกสาร และมีข้อมูลการใช้งานเพื่อวางนโยบาย
  • ตัววัดความสำเร็จ: จำนวนสื่อในคลัง, ยอดดาวน์โหลด/สั่งซื้อ, จำนวนโรงเรียน/ผู้ใช้ที่ active, ความครอบคลุมรายจังหวัด

ผู้ใช้ & บทบาท (Actors)

บทบาททำอะไรได้
ผู้เรียน / นักเรียนค้นหา/อ่าน/ดาวน์โหลดสื่อ e-book ตามสิทธิ์ที่ได้รับ
ครูเลือกรายการหนังสือจากคลัง สร้างคำสั่งซื้อ/เบิกสื่อให้ห้องเรียน
ผู้บริหารโรงเรียนอนุมัติคำสั่งซื้อ ดูประวัติและรายงานระดับโรงเรียน
เจ้าหน้าที่ / ผู้ดูแลระบบ (กสร.)จัดการสิทธิ์ทุกระดับ จัดการคลังสื่อ ดูรายงานภาพรวมรายจังหวัด
โรงพิมพ์ / หน่วยงานภายนอกส่งข้อมูลหนังสือเข้าระบบผ่าน API

2 ขอบเขตงาน (Scope)

แยก in/out ให้ชัด — เกราะกันงานบานปลายและความเข้าใจผิดเรื่องราคา

✅ อยู่ในขอบเขต (In-scope)

  • ระบบจัดการสิทธิ์ผู้ใช้ & บทบาท (RBAC)
  • คลังหนังสืออิเล็กทรอนิกส์ + อัปโหลด/จัดหมวด PDF + PDF viewer
  • API เชื่อมข้อมูลหนังสือจากโรงพิมพ์
  • ระบบสั่งซื้อ/เบิกสื่อ + ประวัติการสั่งซื้อ
  • รายงาน & สถิติ (ยอดดาวน์โหลด, คำสั่งซื้อรายจังหวัด)
  • เชื่อม Single Sign-On (SSO) กับ identity กลางกรม
  • Data pipeline เข้าสู่ Big Data + dashboard วิเคราะห์
  • AI Adaptive Learning (เชื่อม Claude/Gemini/OpenAI + แนะนำสื่อรายบุคคล)

⛔ อยู่นอกขอบเขต (Out-of-scope)

  • เนื้อหา/ลิขสิทธิ์หนังสือ (ลูกค้า/โรงพิมพ์จัดเตรียม)
  • ค่า license/โทเคน AI รายเดือน (Claude/Gemini/OpenAI)
  • ระบบบัญชี/การเงินภายในกรม
  • การพัฒนาระบบปลายทางที่เชื่อม (HRMS, E-Saraban, EKAS ฯลฯ) — ทำเฉพาะ "ส่วนเชื่อมต่อ" ฝั่งเรา
  • Migrate ข้อมูลเก่า หากยังไม่ระบุปริมาณ/รูปแบบ
  • ค่า cloud/hosting/โดเมน & งานดูแลหลังส่งมอบ (เสนอแยก)
หมายเหตุจากเคาะดีญะฮ์: ตามที่พี่ยืนยัน ฉันตีราคา "ทั้ง ecosystem" รวม AI/Big Data แต่ส่วนนี้ความไม่แน่นอนสูง (เป็นงาน R&D) — ฉันจึงแนะนำให้แบ่งเฟส (ดูในส่วนค่าใช้จ่าย)

5 ใช้เวลาทำนานเท่าไร — Next.js vs Laravel

สมมติทีม freelance 3 คน (full-stack 2 + AI/Data 1, PM/QA แชร์กัน · ~2.5 effective heads)

Effort รวมทั้งโครงการ = ~220–380 man-day (รวม contingency 20% แล้ว — ดูที่มาในส่วน 6) เมื่อกระจายให้ทีม ~2.5 คน จะได้ระยะเวลาจริง (elapsed) ประมาณ 5–9 เดือน

▲ Next.js (React / TypeScript)

  • เชื่อม AI / real-time ลื่น (อยู่ใน ecosystem JS เดียวกัน)
  • UX/SPA ทันสมัย เหมาะ dashboard + viewer
  • หา dev รุ่นใหม่ง่าย
  • งาน CRUD/admin ต้องประกอบเองมากกว่า (ช้ากว่าเล็กน้อยในกลุ่ม A)

Elapsed: ~5.5–9 เดือน

▲ Laravel (PHP)

  • CRUD/admin scaffolding เร็ว (Filament/Nova) → กลุ่ม A เร็วกว่า ~10%
  • โตเต็มที่ เหมาะระบบ order/report/admin-heavy
  • ทีมงานภาครัฐไทยคุ้น PHP
  • ส่วน AI/Big Data ยังต้องพึ่ง service Python แยกอยู่ดี

Elapsed: ~5–8.5 เดือน

สรุปตรง ๆ: ผลต่างเวลา/ราคาระหว่างสอง stack เล็กมาก (~5–8%) เพราะกลุ่ม A ที่ Laravel ได้เปรียบ เป็นแค่ ~1/3 ของงาน ส่วน AI/Big Data (กลุ่ม C–D ที่หนักสุด) ต้องใช้ Python microservice เหมือนกันทั้งคู่ → เลือกตามทีมที่พี่หาได้จริงดีกว่าเลือกตามตัวเลข (Laravel ถ้าทีมถนัด PHP, Next.js ถ้าเน้น UX/AI สมัยใหม่)

ฐานการคำนวณ (Basis)

  • 1 man-day = 8 ชั่วโมง · สกุลเงิน บาท (THB) · ยังไม่รวม VAT

7 สมมติฐาน · ความเสี่ยง · คำถามที่ต้องถามลูกค้าเพิ่ม

สิ่งที่ยังไม่ยืนยัน — ตอบครบเมื่อไร ราคาจะแม่นขึ้น

สมมติฐาน (Assumptions)

  • A-01: กรมมีระบบ identity กลางสำหรับ SSO พร้อมเอกสาร API ให้เชื่อม
  • A-02: โรงพิมพ์/หน่วยงานภายนอกมี API หรือไฟล์รูปแบบมาตรฐานให้ดึงข้อมูลหนังสือ
  • A-03: ปริมาณผู้ใช้/ไฟล์ PDF อยู่ในระดับที่ใช้สถาปัตยกรรม cloud มาตรฐานรองรับได้
  • A-04: ใช้ design system / component library สำเร็จรูป ไม่ออกแบบ UI จากศูนย์
  • A-05: ขอบเขต AI = แนะนำสื่อ/เส้นทางเรียนรู้ ไม่ใช่ train โมเดลเอง (เรียกใช้ผ่าน API)

ความเสี่ยง (Risks)

ความเสี่ยงผลกระทบโอกาสแนวทางรับมือ
AI/Big Data requirement ยังไม่นิ่ง (R&D)สูงสูงแยกเป็นเฟส 2, ทำ POC ก่อนล็อกราคา
จำนวน/ความพร้อมของ API ภายนอก (M-07) ไม่ชัดสูงกลางตีราคาเป็นช่วงกว้าง, คิดเป็นรายจุดเชื่อม
การจัดซื้อภาครัฐ + เอกสาร/ขั้นตอนเยอะกลางสูงเผื่อ buffer PM, วาง milestone ชัด
ปริมาณ PDF/ทราฟฟิกโตเกินคาดกลางกลางใช้ object storage + CDN, ออกแบบเผื่อ scale

คำถามที่ต้องถามลูกค้าเพิ่ม (Open Questions)

  • Q-01: จำนวนผู้ใช้/โรงเรียน, ปริมาณ PDF และทราฟฟิกที่คาดไว้?
  • Q-02: SSO ใช้มาตรฐานอะไร (OIDC/SAML/อื่น ๆ) และมีเอกสารพร้อมหรือยัง?
  • Q-03: ต้องเชื่อม API ภายนอกกี่ระบบ และระบบไหนพร้อมในเฟสนี้บ้าง?
  • Q-04: มีข้อมูลเดิมต้อง migrate ไหม ปริมาณเท่าไร รูปแบบใด?
  • Q-05: ขอบเขต "AI Adaptive Learning" ที่ต้องการในเฟสแรกแค่ไหน (แนะนำสื่อ / ทำนายผล / chatbot)?
  • Q-06: งบประมาณกรอบและเดดไลน์ของโครงการ?
