# ระบบคูปอง (Coupon System) — โจทย์ตั้งต้น

> สร้างจาก Inline mode — พี่สั่ง "ช่วยประเมินราคา ระบบคูปอง"
> วันที่: 2026-06-20

## โจทย์ดั้งเดิม
"ช่วยประเมินราคา ระบบคูปอง ให้หน่อย"

## Scope ที่ตกลงกับพี่ (ยืนยันผ่านคำถามเลือกได้)

### รูปแบบคูปอง (ฟีเจอร์หลัก)
- คูปองส่วนลด / โปรโมชัน (%, จำนวนเงิน, ซื้อ 1 แถม 1, เงื่อนไข, วันหมดอายุ, จำกัดจำนวน)
- คูปองแลกของรางวัล (redeem) — มีคลังของรางวัล/สต็อก
- คูปอง QR / ใช้หน้าร้าน — redeem ที่หน้าร้านเท่านั้น

### ช่องทางลูกค้า (end customer)
- **หน้าร้านอย่างเดียว** — ไม่มีระบบ login เต็ม
- ลูกค้าสแกน QR ที่ร้าน → ระบบออก **device-bound session** เก็บใน local เครื่องลูกค้า
- ลูกค้าดูคูปองของตัวเองได้เฉพาะ "ร้านนั้น ๆ"
- **[แก้ตามที่พี่ทักท้วง 2026-06-20]** คูปอง **ไม่หาย** เมื่อเปลี่ยนเครื่อง —
  คูปองเก็บฝั่ง server ผูกกับ **ทะเบียนลูกค้าเบื้องต้น (ชื่อ/เบอร์)** ของแต่ละร้าน;
  สิ่งที่เก็บในเครื่อง = แค่ session (กุญแจเข้าดู)
- เปลี่ยนเครื่อง → **พนักงานร้านค้นหาลูกค้าเดิมแล้วผูกกับเครื่องใหม่ให้** (re-identify by staff)
  → คูปองเดิมกลับมาครบ
- เก็บข้อมูลลูกค้าขั้นต่ำ (ชื่อ/เบอร์) ตอนสแกนครั้งแรก เพื่อให้พนักงานค้นหาได้ — ต้องมี consent PDPA

### โมเดลธุรกิจ & การเชื่อมต่อ
- **Multi-tenant SaaS** — แต่ละร้าน = 1 tenant, เปิด/ปิดร้านได้
- คิด **ค่าบริการรายวัน** ต่อร้าน (ค่า subscription SaaS ปกติ — ไม่มีดอกเบี้ย, halal)
- ชำระเงินด้วย **PromptPay ธรรมดา** — ไม่ผูก bank API / payment gateway
  → ยืนยันยอดแบบ manual (อัปสลิป / แอดมินกดยืนยัน)
- **ไม่เชื่อมระบบภายนอกอื่น** (standalone)

### ขนาดการใช้งาน
- ระดับ **กลาง** — มีผลต่อ non-functional/infra (multi-tenant, backup, uptime)

## สิ่งที่ยังต้องถามเพิ่ม (เพื่อให้ราคาแม่นขึ้น)
- ค่าบริการรายวันต่อร้านคิดเท่าไร / มี tier ไหม
- ลูกค้าปลายทางต้องระบุตัวตน (เบอร์/อีเมล) ไหม หรือ device-anonymous พอ
- จำนวนร้าน + ทรานแซกชันต่อวันที่คาดไว้ (นิยาม "กลาง")
- ต้องมีแอป staff แยกไหม หรือเว็บ responsive สแกนผ่านมือถือพอ
- อนาคตต้อง reconcile PromptPay อัตโนมัติไหม
